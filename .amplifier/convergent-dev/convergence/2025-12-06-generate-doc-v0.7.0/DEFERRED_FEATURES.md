# v0.7.0 Deferred Features

**Release**: v0.7.0  
**Date**: 2025-12-06  
**Deferred Count**: 15+ items

---

## üìã Overview

During the v0.7.0 convergence, we explored two major directions:

1. **"Complete the Workflow"** ‚Üí Update/maintenance features (change detection, selective regen, automation)
2. **"Generate from Scratch"** ‚Üí Forward generation from project analysis (generate-doc)

**Decision**: Focus v0.7.0 on Problem #2 (Generate-Doc) because:
- Solves fundamental "no existing docs" problem
- Enables fresh doc creation workflow
- Rethinks document generation approach
- More foundational than incremental improvements
- User expressed clear vision for this approach

This document explains what was explored and deferred for future releases.

---

## üéØ Major Deferral: "Complete the Workflow" Features

These were the **original focus** before the pivot to generate-doc. All remain valuable for future releases.

### Theme: Update & Regeneration Intelligence

**DE-v0.7.0-001: Selective Section Regeneration**  
**Priority**: P1 - High  
**Effort**: 2-3 days  
**Labels**: `generation`, `selective-update`, `workflow`

**Description:**
Enable regenerating specific sections instead of entire document. User specifies which sections to update (by name or interactively), system regenerates only those sections while preserving others.

**Use Cases:**
```bash
# Regenerate specific sections
doc-evergreen regen --section "Installation"
doc-evergreen regen --section "Usage,Contributing"

# Interactive selection
doc-evergreen regen --interactive
  ‚Üí Shows list, user picks sections
```

**Why Deferred:**
- v0.7.0 focuses on fresh doc generation, not incremental updates
- Requires understanding section boundaries and dependencies
- Build on v0.7.0's nesting-aware generation first

**Reconsider When:**
- v0.7.0 is complete and users want granular update control
- Update workflow becomes primary use case
- User feedback requests this specifically

**Priority for v0.8.0**: P1 (High value for maintenance workflow)

---

**DE-v0.7.0-002: Change Detection & Staleness Awareness**  
**Priority**: P1 - High  
**Effort**: 3-4 days  
**Labels**: `automation`, `change-detection`, `workflow`

**Description:**
Intelligent awareness of when documentation needs updates. Track source file changes, detect staleness, notify users when docs are outdated.

**Approaches Explored:**
- **File Hash Tracking:** Track content hashes of sources, detect changes
- **Git Integration:** Use git history to correlate commits to doc sections
- **Timestamp-Based:** Simple "sources newer than output?"
- **Semantic Change Detection:** LLM analyzes if changes are doc-worthy

**Use Cases:**
```bash
# Check staleness
$ doc-evergreen status README.md
  ‚ö†Ô∏è  Section "API Reference" may be stale
      - src/api.py modified (used by this section)
  ‚úì  All other sections up-to-date

# Regenerate only stale sections
$ doc-evergreen regen --stale-only
```

**Why Deferred:**
- v0.7.0 focuses on initial generation, not maintenance
- Requires tracking generation history and source mappings
- Complex feature, deserves dedicated focus
- Build on v0.7.0's source-section mapping

**Reconsider When:**
- Users have docs generated by doc-evergreen and need update workflow
- Maintenance becomes more important than initial generation
- v0.7.0 source mapping is proven

**Priority for v0.8.0**: P1 (Natural next step after v0.7.0)

---

**DE-v0.7.0-003: Stability Tracking for Sections**  
**Priority**: P2 - Medium  
**Effort**: 2 days  
**Labels**: `generation`, `quality`, `tracking`

**Description:**
Track which sections tend to change vs remain stable during regeneration. Use this to identify sections that need different prompts or sources, or to warn users about unstable sections.

**Why Deferred:**
- Requires multiple regeneration cycles to gather data
- More useful after users have established workflows
- v0.7.0 is about first-time generation

**Reconsider When:**

- v0.8.0 adds regeneration intelligence

**Priority for v0.8.0**: P2

### DE-v0.7.0-010: Bottom-Up Generation Strategy
**Priority**: P3 - Low  
**Effort**: 3-4 days  
**Labels**: `generation`, `enhancement`, `advanced`

**Description:**
Alternative generation strategy: generate deepest sections first, then parents have context from children. More complex than top-down DFS, but might produce better parent sections.

**Why Deferred:**
- Top-down DFS is simpler and sufficient for v0.7.0
- Bottom-up is optimization, not core requirement
- Adds significant complexity
- Prove top-down works first

**Reconsider When:**
- Top-down generation produces poor parent sections
- Parents need context from children to be coherent
- Users request this specifically

**Priority for v0.8.0+**: P3 (Nice-to-have optimization)

**Priority**: P3 - Low  
**Effort**: 5+ days  
**Labels**: `enhancement`, `ml`, `advanced`

Replace LLM-based relevance analysis with trained ML models. Fine-tune embeddings on doc-source relationships, faster inference, potentially better accuracy.

## üéØ Why These Deferrals Make Sense

### Focus: Generate-Doc (Forward Generation)

**v0.7.0 solves ONE problem excellently:** "I have NO docs, create them from project analysis"

**Key decisions:**
1. **Update workflow deferred** ‚Üí Change detection, selective regen, staleness awareness are valuable but orthogonal
2. **Automation deferred** ‚Üí Manual workflow needs to work and be proven first
3. **Enhancements deferred** ‚Üí Core generation needs to work before optimizing
4. **Quality/validation deferred** ‚Üí Learn what quality issues emerge from usage

### Foundation for Future Work

**v0.7.0 enables:**
- v0.8.0: Update/maintenance workflow (change detection, selective regen)
- v0.9.0: Automation & integration (git hooks, CI/CD, watch mode)
- v1.0.0: Advanced features (multi-doc, learning, UI)

**Not deferred because they're bad** ‚Üí Deferred because v0.7.0 has clear, foundational focus.
